6.4 Atkļūdošana un profilēšana

Tavs šīs stundas izaicinājums: Apgūt Godot debugger, breakpoints, un izmantot to, lai atrastu bugs.

SR 2.4.10. Atkļūdošana SR 2.4.19. Algoritmi

Priekšzināšanu robeža: uzdevumos izmanto tikai iepriekš apgūto un šīs lapas teorijas/koda piemēros parādīto. Ja parādās jauna komanda vai rīks, vispirms atrod tās paraugu teorijas blokā.

Teorija: Debugger un profileris

Godot piedāvā vairākus rīkus problēmu atrašanai:

  • Output panel — print() un printerr() ziņojumi.
  • Debugger — breakpoints, step through, watch variables.
  • Profiler — funkciju laika analīze.
  • Network Profiler — RPC un networking.
  • Visual Profiler — vizuāls diagrammu uzziņa.
// Debugging C++ kodā
UtilityFunctions::print("Player position: ", get_position());
UtilityFunctions::printerr("Error: enemy is null!");
UtilityFunctions::push_warning("Suspicious value: ", value);

// Asserts
ERR_FAIL_NULL_MSG(player, "Player not found!");
ERR_FAIL_COND_MSG(hp < 0, "Negative HP!");

// Stack trace dump (debug builds)
#ifdef DEBUG_ENABLED
WARN_PRINT("Function called: " + String(__FUNCTION__));
#endif

Teorijas paplašinājums: C++ fokuss

Kritiskais fokuss: šajā stundā nepietiek tikai panākt redzamu efektu editorā. Jāspēj paskaidrot, kura C++ klase glabā stāvokli, kura metode to maina un kā Godot node struktūra šo kodu izmanto.

Pārbaudes pierādījums: UI signāli, audio darbības, debug izvade un publicēšanas sagatave ir sasaistīta ar C++ spēles stāvokli.

Konsolidētais praktiskais uzdevums (15+ soļi)

70 min darba sadalījums: 1. uzdevums (~20 min) — izveido minimālo piemēru, kas pārbauda šīs stundas galveno ideju; 2. uzdevums (~25 min) — integrē risinājumu tēmas spēles projektā; 3. uzdevums (~25 min) — notestē robežgadījumus, salabo kļūdas un pieraksti secinājumu README vai komentārā. Tālāk redzamie soļi ir detalizēta karte šiem trim uzdevumiem; ja laiks beidzas, papildu pulēšanu atstāj nākamajai reizei.

Izpildi šo darba plānu, lai 6.4 Atkļūdošana un profilēšana rezultāts būtu pārbaudāms C++ GDExtension projektā.

Izpildes soļi:

  1. Atver projektu un izveido Git branch UI, audio, debug vai publicēšanas darbam.
  2. Pieraksti lietotāja ceļu: ko spēlētājs redz, nospiež, dzird vai lejupielādē.
  3. Pārbaudi, ka galvenā spēles stāvokļa loģika joprojām ir C++ klasēs.
  4. Izveido vai atjaunini C++ klasi, kas savieno UI/audio/publicēšanas stāvokli ar spēli.
  5. Ja lieto UI pogas, pieslēdz signālus ar Callable vai reģistrētu C++ metodi.
  6. Ja lieto audio, nosauc busus un skaņu tipus tā, lai tos var saprast pēc nosaukuma.
  7. Ja lieto animācijas, nodali vizuālo efektu no spēles noteikumu aprēķina.
  8. Pievieno debug izvadi tikai vietās, kur tā palīdz atrast stāvokļa kļūdu.
  9. Pārbaudi kļūdu scenāriju: trūkst audio, nav save faila, poga nav pieslēgta vai export fails nav izveidots.
  10. Pārkompilē C++ kodu ar SCons pirms testēšanas Godot vidē.
  11. Palaid spēli no sākuma ekrāna līdz galvenajai darbībai, neizlaižot UI ceļu.
  12. Pārbaudi skaļumu, pauzi vai iestatījumus vismaz divas reizes vienā sesijā.
  13. Ja šī ir eksporta stunda, izveido tīru build mapi un eksportē projektu no jauna.
  14. Pārbaudi, ka eksportā iekļauta GDExtension bibliotēka vajadzīgajai platformai.
  15. Pārbaudi gala rezultātu ārpus Godot editora, piemēram, palaižot eksportēto failu.
  16. README pieraksti lietotāja vadību, zināmos ierobežojumus un build/publish soļus.
  17. Noņem lieku debug izvadi, kas spēlētājam nav vajadzīga.
  18. Veic commit ar ziņu, kas nosauc UI, audio, debug vai publicēšanas izmaiņu.

Aktivitāte 1: Print debugging

Strategic print statements.

Izpildes soļi:

  1. Spēlē izvieto print() svarīgākajās vietās: spawn, damage, level change.
  2. Lieto printerr() warnings un kļūdām.
  3. Pievieno conditional debugging: #ifdef DEBUG_ENABLED.
  4. Pārbaudi Output paneli pēc spēles palaišanas.

Aktivitāte 2: Breakpoints

Izpildes apstāšana.

Izpildes soļi:

  1. VS Code: noklikšķini line number — sarkans punkts = breakpoint.
  2. Lai breakpoints strādā, kompilē ar debug symbols.
  3. F5 (debug run), kad sasniegts breakpoint — programma apstājas.
  4. Step over (F10), step into (F11), continue (F5).

Aktivitāte 3: Profiler analīze

Atrod lēnos kodus.

Izpildes soļi:

  1. Debugger → Profiler → Start.
  2. Spēlē 30 sec, tad Stop.
  3. Skatās top 5 funkcijas pēc CPU laika.
  4. Optimizē tās — re-profile, salīdzini.
  5. Pieraksti uzlabojumus README failā.

Papildu izaicinājums: Memory profiler

Atrod memory leaks.

Izpildes soļi:

  1. Godot debugger → Monitor → Memory.
  2. Vēro, vai memory pieaug spēles laikā (leak indicator).
  3. Lieto Valgrind (Linux) sistēmas līmenī.

Biežākās kļūdas

  • Breakpoint nestrādā: Debug build trūkst — kompilē ar scons target=template_debug.
  • Print panel pārplūst: Pārāk daudz print — lieto conditional vai noņem.
  • Profiler crashes: Pārliecinies par jaunāko Godot versiju.

Godot ekrānuzņēmumi

Koda piemērs (paplašināts)

// Defensīva programmēšana
void Player::take_damage(int amount) {
    // Validate input
    ERR_FAIL_COND_MSG(amount < 0,
        "Damage amount must be non-negative, got: " + String::num_int64(amount));

    // Log for debugging
    UtilityFunctions::print("Player ", name, " taking ", amount,
                            " damage. Current HP: ", hp);

    // Apply damage
    hp = std::max(0, hp - amount);

    // Sanity check
    if (hp < 0) {
        UtilityFunctions::printerr("CRITICAL: HP went negative! Resetting to 0");
        hp = 0;
    }

    // Conditional debug
    #ifdef DEBUG_ENABLED
    if (hp == 0) {
        UtilityFunctions::print("Player died at position ", get_position());
    }
    #endif

    emit_signal("damaged", amount);
}
Output panelī redzami detalizēti debug ziņojumi; pie nederīgas ievades — error ar context.
