3.6 Noslēguma projekts: RPG cīņas spēle

Tavs šīs stundas izaicinājums: Apvienot OOP, signālus, struct un GDExtension klases pilnvērtīgā rundām balstītā RPG cīņā.

SR 2.4.1. Problēmu dekompozīcija SR 2.4.7. UI SR 2.4.15. Modulāras programmas SR 2.4.18. OOP

Priekšzināšanu robeža: uzdevumos izmanto tikai iepriekš apgūto un šīs lapas teorijas/koda piemēros parādīto. Ja parādās jauna komanda vai rīks, vispirms atrod tās paraugu teorijas blokā.

Teorija: Spēles apraksts

Rundām balstīta RPG cīņa ar trim spēlētāja varoņiem (Knight, Mage, Archer) pret nejauši ģenerētiem ienaidniekiem (Goblins, Pūķis, Skelets).

Spēles plūsma:

  1. Spēles sākumā izvēlies 3 varoņus partu.
  2. Ienaidnieki tiek ģenerēti.
  3. Katras kārtas: spēlētājs izvēlas darbību (uzbrukt, burvojums, dziedināt).
  4. Pēc tam ienaidnieki uzbrūk.
  5. Cīņa turpinās, līdz viena puse ir mirusi.

Klašu hierarhija: Hero (base) → Knight, Mage, Archer; Enemy (base) → Goblins, Pūķis, Skelets.

Teorijas paplašinājums: C++ fokuss

Kritiskais fokuss: šajā stundā nepietiek tikai panākt redzamu efektu editorā. Jāspēj paskaidrot, kura C++ klase glabā stāvokli, kura metode to maina un kā Godot node struktūra šo kodu izmanto.

Pārbaudes pierādījums: katrai būtiskai spēles lomai ir C++ klase ar header/implementation dalījumu, konstruktoriem, metodēm, signāliem vai mantošanu.

Konsolidētais praktiskais uzdevums (15+ soļi)

70 min darba sadalījums: 1. uzdevums (~20 min) — izveido minimālo piemēru, kas pārbauda šīs stundas galveno ideju; 2. uzdevums (~25 min) — integrē risinājumu tēmas spēles projektā; 3. uzdevums (~25 min) — notestē robežgadījumus, salabo kļūdas un pieraksti secinājumu README vai komentārā. Tālāk redzamie soļi ir detalizēta karte šiem trim uzdevumiem; ja laiks beidzas, papildu pulēšanu atstāj nākamajai reizei.

Izpildi šo darba plānu, lai 3.6 Noslēguma projekts: RPG cīņas spēle rezultāts būtu pārbaudāms C++ GDExtension projektā.

Izpildes soļi:

  1. Atver projektu un izveido Git branch šīs stundas OOP darbam.
  2. Apraksti vienā teikumā, kādu spēles lomu modelē jaunā C++ klase.
  3. Izveido .hpp failu ar klases deklarāciju un privātajiem datiem.
  4. Izveido .cpp failu ar konstruktoru un metožu implementāciju.
  5. Pārliecinies, ka katrai klasei ir viena skaidra atbildība.
  6. Ja klase ir Godot node, pievieno GDCLASS makro un _bind_methods().
  7. Ja klase ir parasta C++ palīgklase, atstāj to neatkarīgu no Scene Tree.
  8. Pievieno konstruktoru, kas objektu atstāj derīgā sākumstāvoklī.
  9. Pievieno vismaz vienu metodi, kas maina objekta stāvokli kontrolētā veidā.
  10. Pievieno vismaz vienu const metodi, kas tikai nolasa stāvokli.
  11. Ja lieto mantošanu, pārbaudi, ka bāzes klase satur kopīgo uzvedību, nevis konkrētas detaļas.
  12. Ja lieto signālus, deklarē tos _bind_methods() un pieslēdz ar Callable.
  13. Reģistrē Godot redzamās klases register_types.cpp failā.
  14. Pievieno īsu C++ izvadi vai UI pārbaudi, kas pierāda, ka metode tiek izsaukta.
  15. Pārkompilē projektu un izlabo visas kompilatora kļūdas pirms Godot testēšanas.
  16. Godot editorā pārbaudi, vai pielāgotā klase vai tās īpašības parādās pareizajā vietā.
  17. README pieraksti, kāpēc izvēlējies kompozīciju, mantošanu vai signālus.
  18. Veic commit ar ziņu, kas nosauc izveidoto klasi vai OOP principu.

Aktivitāte 1: Klašu hierarhija un Stats (1.)

Pamata struktūra.

Izpildes soļi:

  1. Pārkopē Hero, Knight, Mage, Archer klases no iepriekšējām stundām.
  2. Tāpat Enemy bāzes klase un 3 apakšklases.
  3. Visiem Stats struct caur kompozīciju.
  4. BattleManager klase: pārvalda kārtas un secību.

Aktivitāte 2: Cīņas mehānika (2.)

Kārtu sistēma.

Izpildes soļi:

  1. BattleManager glabā std::vector<Hero*> party un std::vector<Enemy*> enemies.
  2. start_turn(): parāda darbību izvēles UI.
  3. Spēlētājs izvēlas darbību → execute_player_action().
  4. Pēc tam execute_enemy_actions(): katrs ienaidnieks uzbrūk nejaušam Hero.
  5. Pārbauda uzvaru/zaudēšanu pēc katras kārtas.

Aktivitāte 3: UI un signāli (3.)

Polish un finalizācija.

Izpildes soļi:

  1. BattleUI scēna ar HP bāriem visiem dalībniekiem.
  2. Hero/Enemy emit damaged/died signālus, BattleUI tos konnekt.
  3. Action menu: 4 pogas (Attack, Cast, Heal, Run).
  4. Damage numbers — populāri attēlojumi, kas izpeld no rakstura.
  5. GitHub: 6+ commit, README, varbūt arī screenshots/GIF.

Papildu izaicinājums: Bonus: Boss cīņa

Specializēts ienaidnieks.

Izpildes soļi:

  1. BossEnemy : public Enemy ar 500 HP un 3 spējām.
  2. Speciālas mehānikas: katras 3 kārtas izmanto super atāku.
  3. Pievienots pēc 3 normālām cīņām.

Biežākās kļūdas

  • Memory leak: 'new' bez 'delete'. Lieto std::unique_ptr vai Godot's add_child (kas pārvalda lifetime).
  • UI nesinhronizējas: Pievienojot signal connect tikai vienā vietā, izveido visu HP bar update sistēmu konsistentu.
  • Cīņa nebeidzas: Pārbaudi nāves apstrādes loģiku — vai tiešām visi mirdzi tiek noņemti no vector.

Godot ekrānuzņēmumi

Koda piemērs (paplašināts)

class BattleManager : public Node {
    GDCLASS(BattleManager, Node)
private:
    std::vector<Hero*> party;
    std::vector<Enemy*> enemies;
    int current_turn = 0;

public:
    void start_battle() {
        for (Enemy* e : enemies) {
            e->connect("died", Callable(this, "on_enemy_died"));
        }
        next_turn();
    }

    void execute_attack(Hero* attacker, Enemy* target) {
        int damage = attacker->get_stats().get_total_attack();
        target->take_damage(damage);
        if (enemies.empty()) emit_signal("battle_won");
        else next_turn();
    }

    void on_enemy_died() {
        // Atrod un noņem mirušo
        enemies.erase(...);
    }
};
