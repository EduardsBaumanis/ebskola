5.3 Save/Load arhitektūra

Tavs šīs stundas izaicinājums: Izveidot pilnvērtīgu saglabāšanas sistēmu ar vairākiem slotiem un autosave.

SR 2.4.17. Failu I/O SR 2.4.6. Plāno izstrādes procesu

Priekšzināšanu robeža: uzdevumos izmanto tikai iepriekš apgūto un šīs lapas teorijas/koda piemēros parādīto. Ja parādās jauna komanda vai rīks, vispirms atrod tās paraugu teorijas blokā.

Teorija: Save sistēmas arhitektūra

Profesionāla save sistēma sastāv no vairākiem komponentiem:

  • SaveManager — galvenais koordinators.
  • Saveable interfeiss — visi saglabājami objekti to implementē.
  • Save slots — vairāki saglabāšanas faili (slot1.json, slot2.json, ...).
  • Autosave — automātiska saglabāšana ik pa N minūtēm vai pie checkpoint.
// Saveable interface
class Saveable {
public:
    virtual Dictionary save_state() const = 0;
    virtual void load_state(const Dictionary& state) = 0;
};

class SaveManager : public Node {
    std::vector<Saveable*> saveables;
public:
    void register_saveable(Saveable* s) {
        saveables.push_back(s);
    }
    void save_to_slot(int slot);
    void load_from_slot(int slot);
};

Konsolidētais praktiskais uzdevums (15+ soļi)

70 min darba sadalījums: 1. uzdevums (~20 min) — izveido minimālo piemēru, kas pārbauda šīs stundas galveno ideju; 2. uzdevums (~25 min) — integrē risinājumu tēmas spēles projektā; 3. uzdevums (~25 min) — notestē robežgadījumus, salabo kļūdas un pieraksti secinājumu README vai komentārā. Tālāk redzamie soļi ir detalizēta karte šiem trim uzdevumiem; ja laiks beidzas, papildu pulēšanu atstāj nākamajai reizei.

Izpildi šo darba plānu, lai 5.3 Save/Load arhitektūra rezultāts būtu pārbaudāms C++ GDExtension projektā.

Izpildes soļi:

  1. Atver projektu un izveido Git branch datu pastāvības vai algoritmu darbam.
  2. Pieraksti, kādi dati jāsaglabā, jāielādē vai jāģenerē šajā stundā.
  3. Izveido C++ klasi, kas atbild par šo datu plūsmu, piemēram, SaveManager vai Generator.
  4. Definē datu modeli ar skaidriem tipiem: int, float, String, Dictionary vai pielāgotu struct.
  5. Pievieno versijas lauku, ja dati tiks saglabāti failā.
  6. Rakstīšanas metodē pārbaudi, vai fails tiešām ir atvērts rakstīšanai.
  7. Lasīšanas metodē pārbaudi, vai fails eksistē un nav tukšs.
  8. Validē ielādētos laukus pirms to izmantošanas spēles stāvoklī.
  9. Kļūdas gadījumā atgriez drošu noklusējuma vērtību, nevis ļauj spēlei sabrukt.
  10. Ja tēma ir algoritms, pieraksti aptuveno sarežģītību, piemēram, O(n) vai O(n log n).
  11. Pievieno mazu testdatu piemēru, kura rezultātu vari pārbaudīt ar roku.
  12. Pievieno lielāku testu, lai pamanītu veiktspējas vai atmiņas problēmas.
  13. Pievieno UtilityFunctions::print izvadi tikai svarīgiem debug punktiem.
  14. Pārkompilē C++ kodu ar SCons un izlabo kompilācijas kļūdas.
  15. Palaid Godot un testē saglabāšanu, ielādi vai ģenerāciju vismaz divas reizes pēc kārtas.
  16. README pieraksti failu ceļus, datu formātu un zināmos ierobežojumus.
  17. Pārbaudi, ka ģenerētie vai saglabātie faili nav nejauši pievienoti Git, ja tie ir lokāli dati.
  18. Veic commit ar ziņu, kas nosauc saglabāšanas, ielādes vai algoritma uzlabojumu.

Aktivitāte 1: Saveable interfeiss

Padari objektus saglabājamus.

Izpildes soļi:

  1. Definē class Saveable ar pure virtual save_state un load_state.
  2. Player, Enemy, Inventory, Quest klases manto no Saveable.
  3. Katra implementē save_state -> Dictionary un load_state -> void.
  4. SaveManager iziet caur visiem un apkopo Dictionaries.

Aktivitāte 2: Vairāki save slots

3 atsevišķi save faili.

Izpildes soļi:

  1. UI: 3 slot pogas ar metadatu (datumu, level).
  2. save_to_slot(int n): fails user://save_+n+.json.
  3. Mainīga thumbnail/screenshot pievienoti.
  4. Iespēja dzēst atsevišķus slots.

Aktivitāte 3: Autosave

Automātiska saglabāšana.

Izpildes soļi:

  1. Timer ik pa 5 min — saglabā uz autosave slot.
  2. Pie checkpoint — manuāla autosave.
  3. Pirms dzēšanas — backup vecākam.
  4. Statusa rāda spēles stūrī "Saving..."

Papildu izaicinājums: Cloud save

Steam Cloud vai citu sync.

Izpildes soļi:

  1. Eksperimentāli — Godot piedāvā Steam integrāciju.
  2. Konfigurē saglabāšanas mapes sinhronizāciju.
  3. Konflikta risināšana, ja vairākas ierīces.

Biežākās kļūdas

  • Save fails fragmented: Save daļas atsevišķos failos vs viens — izvēlies vienu pieeju.
  • Race condition: Save un load notiek vienlaikus — lieto mutex vai sekvenciāli.
  • Slot UI nesinhronizējas: Pārbaudi metadata refresh pēc save.

Godot ekrānuzņēmumi

Koda piemērs (paplašināts)

class SaveManager : public Node {
    GDCLASS(SaveManager, Node)
private:
    std::vector<Node*> saveables;
    float autosave_timer = 0;

public:
    void _process(double delta) override {
        autosave_timer += delta;
        if (autosave_timer >= 300.0f) {  // 5 min
            save_to_slot(0);  // 0 = autosave
            autosave_timer = 0;
        }
    }

    void save_to_slot(int slot) {
        Dictionary all_data;
        for (Node* n : saveables) {
            if (n->has_method("save_state")) {
                all_data[n->get_path()] = n->call("save_state");
            }
        }
        all_data["timestamp"] = Time::get_singleton()->get_unix_time_from_system();
        all_data["version"] = 1;

        String path = "user://save_" + String::num_int64(slot) + ".json";
        Ref<FileAccess> f = FileAccess::open(path, FileAccess::WRITE);
        f->store_string(JSON::stringify(all_data, "  "));
        f->close();
    }
};
3 save slot ar metadatu (level, time played, last save), autosave ik 5 min, korekta load.
