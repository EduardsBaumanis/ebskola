2.3 Vadības struktūras spēles loģikā

Tavs šīs stundas izaicinājums: Izmantot if/else, switch un cikli spēles uzvedības loģikā.

SR 2.4.14. Vadības struktūras SR 2.4.13. Mainīgie

Priekšzināšanu robeža: uzdevumos izmanto tikai iepriekš apgūto un šīs lapas teorijas/koda piemēros parādīto. Ja parādās jauna komanda vai rīks, vispirms atrod tās paraugu teorijas blokā.

Teorija: C++ vadības struktūras

C++ vadības struktūras ir gandrīz identiskas Python ekvivalentiem, bet ar atšķirīgu sintaksi:

// If / else if / else
if (player->health <= 0) {
    player->die();
} else if (player->health < 30) {
    play_warning_sound();
} else {
    show_normal_status();
}

// Switch (kā Python match-case)
switch (state) {
    case PlayerState::IDLE:    play_idle();    break;
    case PlayerState::RUNNING: play_run();     break;
    case PlayerState::JUMPING: play_jump();    break;
    default:                   break;
}

// For cikls
for (int i = 0; i < enemies.size(); i++) {
    enemies[i]->update();
}

// Range-based for (C++11+)
for (Enemy* e : enemies) {
    e->update();
}

// While
while (health > 0 && level_active) {
    process_combat();
}

Teorijas paplašinājums: C++ fokuss

Kritiskais fokuss: šajā stundā nepietiek tikai panākt redzamu efektu editorā. Jāspēj paskaidrot, kura C++ klase glabā stāvokli, kura metode to maina un kā Godot node struktūra šo kodu izmanto.

Pārbaudes pierādījums: spēlētājs, objekti un sadursmes darbojas ar C++ klasēm, kas lieto _physics_process, Input, CharacterBody2D, Area2D vai move_and_slide.

Konsolidētais praktiskais uzdevums (15+ soļi)

70 min darba sadalījums: 1. uzdevums (~20 min) — izveido minimālo piemēru, kas pārbauda šīs stundas galveno ideju; 2. uzdevums (~25 min) — integrē risinājumu tēmas spēles projektā; 3. uzdevums (~25 min) — notestē robežgadījumus, salabo kļūdas un pieraksti secinājumu README vai komentārā. Tālāk redzamie soļi ir detalizēta karte šiem trim uzdevumiem; ja laiks beidzas, papildu pulēšanu atstāj nākamajai reizei.

Izpildi šo darba plānu, lai 2.3 Vadības struktūras spēles loģikā rezultāts būtu pārbaudāms C++ GDExtension projektā.

Izpildes soļi:

  1. Atver esošo Godot projektu un izveido jaunu Git branch šīs stundas darbam.
  2. Pārbaudi, ka spēles loģika tiek rakstīta C++ GDExtension klasē.
  3. Pieraksti, kurš node tips šajā stundā ir galvenais: Node2D, CharacterBody2D, Area2D vai RigidBody2D.
  4. Izveido vai atjaunini atbilstošu .hpp failu ar klases deklarāciju.
  5. Izveido vai atjaunini atbilstošu .cpp failu ar metožu implementāciju.
  6. Pievieno nepieciešamos #include failus no godot_cpp.
  7. Pievieno GDCLASS un tukšu vai vajadzīgu _bind_methods() metodi.
  8. Reģistrē klasi register_types.cpp ar ClassDB::register_class.
  9. Kustībai lieto _physics_process(double delta), nevis tikai manuālu pozīcijas maiņu editorā.
  10. Ja maini pozīciju, reizinājumā iekļauj delta, lai ātrums nebūtu atkarīgs no FPS.
  11. Ja objekts saduras ar vidi, izvēlies move_and_slide() vai move_and_collide() un pamato izvēli.
  12. Iestati CollisionShape2D izmēru tā, lai tas sakristu ar redzamo objektu.
  13. Pārbaudi collision layer/mask, lai objekts reaģē tikai uz vajadzīgajiem slāņiem.
  14. Pievieno īsu UtilityFunctions::print pārbaudi, kas parāda svarīgu stāvokli spēles laikā.
  15. Pārkompilē ar SCons un pārliecinies, ka Godot ielādē jauno bibliotēku bez kļūdām.
  16. Testē vienu normālu gadījumu un vienu robežgadījumu, piemēram, ekrāna malu vai platformas malu.
  17. README pieraksti, kāda fizikas vai sadursmju problēma tika atrisināta.
  18. Veic commit ar skaidru ziņu par šīs stundas mehāniku.

Aktivitāte 1: Veselības sistēma

Implementē varoņa veselības stāvokļus.

Izpildes soļi:

  1. Player klasei pievieno int health = 100, int max_health = 100.
  2. Metode take_damage(int amount): health = std::max(0, health - amount);
  3. Ja health == 0: queue_free() (pārtrauc spēli vai restart).
  4. Ja health < 30%: izvadi brīdinājumu Output panelī.
  5. Pievieno HUD Label, kas rāda HP.

Aktivitāte 2: Punktu sistēma ar switch

Dažādu objektu vērtība.

Izpildes soļi:

  1. Pievieno enum: enum class PickupType { COIN, GEM, KEY };
  2. Funkcija void on_pickup(PickupType type) ar switch.
  3. COIN += 1 punkts; GEM += 5; KEY = atver durvis.
  4. Izveido 3 dažādus Area2D ar Pickup klasi katrā tipā.
  5. Spēlētājs paaugstina punktus, savācot.

Aktivitāte 3: Cikli ienaidnieku pārbaudei

Pārbaudi visus ienaidniekus.

Izpildes soļi:

  1. Pievieno std::vector<Enemy*> enemies Game klasē.
  2. _physics_process: ar for cikls iet caur visiem un pārbauda, vai kāds ir mirdzs (health==0).
  3. Mirdzos noņem no saraksta: enemies.erase(...).
  4. Ja saraksts tukšs — uzvara!

Papildu izaicinājums: Bonus: Range-based for

Modernā C++ sintakse.

Izpildes soļi:

  1. Pārveido visus for ciklus uz range-based: for (auto& e : enemies)
  2. Salīdzini koda salasāmību.
  3. Iemēģini auto automātisku tipu noteikšanu.

Biežākās kļūdas

  • 'enemies' was not declared: Trūkst #include <vector> un namespace.
  • Crash, mēģinot dzēst no vektora cikla laikā: Lieto pareizu iterator pieeju vai dzēš pēc cikla.
  • switch case nestrādā: Aizmirsts break; — kods iet uz nākamo case.

Godot ekrānuzņēmumi

Koda piemērs (paplašināts)

enum class PickupType { COIN, GEM, KEY };

void Player::on_pickup(PickupType type) {
    switch (type) {
        case PickupType::COIN:
            score += 1;
            UtilityFunctions::print("Coin! Score: ", score);
            break;
        case PickupType::GEM:
            score += 5;
            UtilityFunctions::print("Gem! Score: ", score);
            break;
        case PickupType::KEY:
            has_key = true;
            UtilityFunctions::print("Got the key!");
            break;
    }

    if (score >= 100) {
        UtilityFunctions::print("Level complete!");
    }
}
Spēlētājs savāc 3 dažādus tipus pickup objektus, score atjauninās terminālī.
