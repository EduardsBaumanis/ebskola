# Pedagogy and Philosophy Essays on the Three HTML Lesson Courses

Source corpus reviewed: `datorika9/`, `programmesana1/`, and `programmesana2/`. The essays below are written from four different educational-philosophical viewpoints and are grounded in the course arcs: Datorika 9 moves from web structure, design thinking, CSS, multimedia, JavaScript, and browser game publication; Programmēšana I moves from Python tooling, GitHub, input/output, validation, modularity, collections, data persistence, algorithms, web integration, databases, OOP, and deployment; Programmēšana II moves from Godot and C++ through physics, OOP, AI, persistence, procedural generation, UI, audio, debugging, profiling, export, and publication.

## Essay 1: The Workshop Curriculum, a Pragmatist and Constructivist Reading

The three-course sequence can be read as a deliberately practical philosophy of knowledge. It does not treat computing as a body of definitions to be transferred into the student before practice begins. Instead, it treats knowing as something that becomes visible through making, revising, testing, documenting, and presenting public artifacts. Datorika 9 begins with the relatively concrete world of the browser: HTML structure, CSS style, media files, JavaScript interactivity, event listeners, DOM manipulation, timers, collision detection, and finally a published game. Programmēšana I then moves the learner into Python, GitHub, modular code, files, data analysis, algorithms, databases, OOP, REST APIs, Flask, and cloud deployment. Programmēšana II intensifies the same logic in Godot 4 and C++ GDExtension, where the learner must connect editor-visible objects with compiled code, physics cycles, state machines, pathfinding, object pooling, save/load architecture, profiling, UI, audio, and public export. In the pragmatist tradition of John Dewey, this is a curriculum of occupations: meaningful work that reorganizes experience into more intelligent future action.

The central pedagogical strength of this design is that concepts arrive as answers to felt problems. HTML is not merely a markup language; it is the skeleton of a game page. CSS is not merely syntax; it is the visual ergonomics through which a player understands an interface. JavaScript variables are not inert examples; they become scores, lives, choices, arrays of game data, and conditions for interaction. Python input, conversion, arithmetic, loops, and validation are not isolated exercises; they become the "Lielais skaitļu duelītis," "Gudrais Akmens-Šķēres-Papīrīts," a number guessing game, quizzes, highscore systems, and browser-based tic-tac-toe. In Programmēšana II, the abstractions are more demanding, but the structure remains recognizably Deweyan: a theory becomes meaningful when it helps the student control a situation. A state machine matters because an enemy needs coherent AI behavior; object pooling matters because repeated instantiation can damage performance; JSON serialization matters because a player's progress should survive beyond the current session.

This is also a constructionist curriculum in Seymour Papert's sense. Students are not only solving teacher-made worksheets; they are producing artifacts that can be inspected, played, shared, and defended. The repeated final projects are especially important. Datorika 9 culminates in a full internet-published game. Programmēšana I culminates repeatedly in increasingly robust software products: terminal games, memory cards, quiz systems, a highscore database, a class tournament simulator, and a multiplayer web game. Programmēšana II culminates each theme in a genre project: Pong, platformer, tournament simulator, top-down shooter, procedural maze, and puzzle game on the web. The public or semi-public artifact changes the epistemology of the course. A student can no longer claim understanding only by repeating an explanation. The code must run, the interface must communicate, the repository must be organized, and the documentation must help someone else enter the project.

The lesson pattern reinforces this constructionist epistemology. Across the courses, many lessons include a brief theory section, practical tasks, common errors, code examples, and often a 70-minute work division. This pattern creates a scaffold rather than a script. The learner is invited to rebuild a minimal example, adapt it to the main task, test the result, correct mistakes, and then write a conclusion. That rhythm is pedagogically significant. It converts the classroom hour from a lecture container into an inquiry cycle. First, the student encounters a model; second, the student acts on it; third, the student tests what the action produced; fourth, the student articulates what worked and what needs repair. In Vygotskian terms, the lessons operate inside the zone of proximal development: the example and the teacher's structure make a task possible before it is fully independent, while the final project gradually withdraws some of that support.

The curriculum also uses spiral progression well. Datorika 9 introduces GitHub integration, file organization, design research, UX/UI, JavaScript logic, testing, and publication. Programmēšana I returns to many of those same ideas at a more formal level: GitHub and README files become normal professional habits; user feedback becomes Google Forms and data analysis; validation becomes try/except; design planning becomes user stories, ER diagrams, UI flow, pitch preparation, and resource planning; deployment becomes Render or Railway; web communication becomes REST and JSON. Programmēšana II returns again, but now within a more industrial toolchain. The student must understand build compatibility, C++ class registration, Godot nodes, physics loops, editor integration, profiling, and export. The recurrence is not repetition in a weak sense. It is the repeated encounter with a family of ideas under new constraints. That is exactly how durable understanding is formed.

From a pragmatist standpoint, the sequence wisely refuses the false separation between "thinking" and "doing." Planning is treated as part of programming, not as decoration before programming. This is visible in Datorika 9, where game concept, target audience, design research, technical documentation, and prototype planning precede the full implementation. It is visible in Programmēšana I, where algorithms are planned in comments, projects are documented in README files, modularity is discussed as a response to complexity, and a whole theme is devoted to project design before implementation. It is visible again in Programmēšana II, where each theme warns against superficial editor-clicking and insists that the game logic must be realized in C++ classes. The philosophical message is clear: technical action is intelligent only when it is planned, justified, tested, and revised.

A further pragmatist strength is the way the courses connect different kinds of knowledge without forcing them into artificial compartments. The Datorika 9 sequence makes design, media, language, file systems, algorithms, and presentation part of the same project. Programmēšana I connects mathematics, language, documentation, law, data analysis, systems architecture, and communication. Programmēšana II connects physics, spatial reasoning, performance, audio, interface design, and software engineering. In a traditional subject-divided school, these relations can become invisible. Here they become practical necessities. A student cannot finish a browser game by knowing only syntax; the student must also make aesthetic decisions, manage assets, explain rules, and consider how another person will experience the product. This integration realizes Dewey's argument that school knowledge should not be a set of detached subject fragments, but a reconstruction of experience through purposeful activity.

The role of the teacher in such a curriculum is therefore closer to a studio mentor than a lecturer or examiner. The teacher introduces concepts, demonstrates tools, notices misconceptions, and protects the inquiry from becoming either chaos or mechanical copying. In Datorika 9, a teacher may help a student see that a game concept is too large for the available time. In Programmēšana I, the teacher may ask whether a function really has one responsibility, whether a dictionary is more appropriate than a list, or whether a database schema can represent the needed relationships. In Programmēšana II, the teacher may press the student to explain why a state machine is cleaner than scattered conditionals, or why object pooling is justified in a particular mechanic. These interventions are not departures from project-based learning. They are the intellectual guidance that makes project-based learning rigorous.

Assessment, from this viewpoint, should be understood as inquiry into growth rather than only measurement of output. The courses contain natural evidence for such assessment: repository history, README quality, planning documents, user stories, diagrams, code organization, test notes, debugging records, project defenses, and published artifacts. A pragmatist assessment system would ask students to narrate how their understanding changed. What did the first prototype reveal? Which assumption failed? What did user feedback change? Which bug taught a transferable lesson? Which tool was chosen, and why? This protects the curriculum from a shallow product culture in which the most polished game automatically appears to be the best learning. The best educational artifact is not always the flashiest one; it is the one in which the student can show intelligent reconstruction of experience.

The courses also work well because they repeatedly introduce threshold concepts: ideas that are troublesome at first, but reorganize what the learner can see once understood. In Datorika 9, the DOM changes a web page from a fixed document into a manipulable environment. In Programmēšana I, functions, files, dictionaries, Big O, relational keys, and APIs each change the student's sense of what a program can be. In Programmēšana II, the Godot Scene Tree, physics cycle, C++ class registration, signals, state machines, and profiling similarly open new forms of perception. A pragmatist curriculum should not rush through such concepts as vocabulary. It should give students occasions to experience the difference they make. The reviewed lessons often do this by embedding the concept inside a concrete need: a timer because the game must continue over time, a JOIN because data lives in more than one table, a state machine because AI behavior must remain intelligible, a profile because performance claims need evidence.

The same sequence teaches that knowledge is social. GitHub is not used only as a storage service; it is a model of public, traceable work. README files, licenses, commits, folder structure, presentation, peer review, project defense, and publication all place the learner's work before an audience. This matters because computing education can easily become private and opaque: a student alone with a screen, producing code that only the teacher reads. These materials instead encourage the student to imagine a user, a player, a collaborator, a future self, and sometimes a wider public. Such imagination is not an accessory to technical learning. It is part of what technical literacy means in democratic and professional life.

There are, however, risks in a strongly project-based curriculum. The first risk is that the project may seduce the student into performance without conceptual clarity. A playable game can hide shallow understanding if students copy examples without explaining why the code works. The materials anticipate this risk through testing, reflection, common error sections, and project defense, but the teacher's enactment remains crucial. The classroom must ask not only "Does it work?" but also "Why this structure?", "What alternatives were considered?", "What data does this code depend on?", and "What would break if the input changed?" A second risk is that project themes, especially games, may motivate many learners while leaving others outside the imaginative center. The curriculum offsets this somewhat by including user research, surveys, data analysis, web apps, databases, and tools, but teachers should still invite alternative project domains where possible.

Another philosophical tension concerns scaffolding. The repeated structure of "theory, tasks, errors, code example" is powerful, but if it becomes too predictable, it may train compliance rather than inquiry. Pragmatism would defend structure only if it increases freedom for intelligent action. Therefore, the most successful use of these lessons would treat the first and second tasks as guided entry, but the third task and final projects as spaces for genuine student choice. Students should be allowed to choose mechanics, data models, interface decisions, difficulty curves, and presentation strategies. Without such choices, the artifact becomes merely the teacher's product assembled by student hands.

Still, the dominant philosophical achievement of the three-course arc is substantial. It gives students a developmental path from visible web pages to abstract algorithmic thinking, from isolated syntax to systems architecture, from local files to public deployment, from small scripts to game engines, from novice experimentation to disciplined technical judgment. Its pedagogy is not pure discovery learning, nor is it transmission teaching. It is a workshop model: tools are introduced when needed, concepts are named when they clarify practice, and artifacts are used as occasions for reflection. In this sense, the curriculum embodies a mature pragmatist thesis: education is not preparation for life in the abstract; it is a carefully structured form of life in which students already participate in the practices they are learning to understand.

## Essay 2: Code, Agency, and the Politics of Technical Literacy, a Critical Pedagogy Reading

Viewed through critical pedagogy, these courses are not merely about programming. They are about who gets to act in a technological society, who understands the systems that shape everyday life, and who is invited to move from consumption to authorship. A neutral reading might say that Datorika 9 teaches HTML, CSS, JavaScript, file organization, and game publication; Programmēšana I teaches Python, Git, data structures, algorithms, databases, OOP, APIs, and deployment; Programmēšana II teaches Godot and C++ for game development. A critical reading asks a further question: what kind of subject is being formed by these lessons? The answer is promising but not automatic. The courses have the raw materials for emancipatory digital education because they ask students to research users, plan products, document code, interpret data, respect licenses, publish work, and defend design decisions. Yet they also need explicit critical dialogue so that technical competence does not become mere adaptation to existing platforms and market logics.

Paulo Freire's distinction between banking education and problem-posing education is a useful starting point. Banking education deposits knowledge into passive learners. Problem-posing education invites learners to name the world, question it, and transform it. The three-course sequence leans toward problem-posing pedagogy because the student repeatedly confronts a meaningful situation: a game needs a target audience, a website needs structure, a quiz needs persistent scores, a highscore system needs a database, a browser game needs an interface, an AI enemy needs states, a save system needs data validation, a public project needs documentation and deployment. The student is not only answering the teacher's question. The student is designing a small world in which other people will act.

Datorika 9 is especially interesting in this regard because it places design thinking before and beside code. Students investigate game concepts, target audiences, UX/UI, market-like comparison, multimedia assets, ergonomics, and publication. This matters politically. When a learner asks who a game is for, what counts as usable, what images and sounds communicate, and how a project becomes accessible online, the learner begins to see technology as a cultural artifact. A web page or game is not merely technical execution. It is an address to someone. It includes and excludes. It persuades, invites, rewards, frustrates, and sometimes manipulates. Critical pedagogy would encourage the teacher to make those issues explicit: Which players are imagined? What abilities, languages, devices, and tastes are assumed? What would make the interface fairer or more inclusive? What values does the game reward?

Programmēšana I deepens the political field by introducing code management, documentation, licenses, user feedback, data, algorithms, web libraries, databases, APIs, and cloud deployment. The theme on open-source licenses and documentation is more than administrative content. It invites students to understand software as a social and legal commons, not simply as private effort. The work with Google Forms, survey data, CSV, JSON, and visualization similarly opens a path toward data literacy. However, critical pedagogy would insist that data is never innocent. When students collect player opinions, store highscores, or design REST APIs, they should discuss consent, privacy, representation, error, and interpretation. A highscore table is technically simple, but socially meaningful: it ranks players, remembers behavior, and can expose identity. Even in a school game, this is an opportunity to ask what data should be kept, for how long, by whom, and for what purpose.

Programmēšana II raises the stakes again. Godot and C++ GDExtension place learners closer to professional game development practices: physics cycles, input systems, OOP architectures, AI state machines, pathfinding, object pooling, procedural generation, profiling, UI, audio, and export. A purely technical curriculum might celebrate this as advanced skill alone. A critical curriculum would also examine the industry practices and assumptions behind those skills. What does it mean to model behavior as states? What assumptions are embedded in enemy AI? How do difficulty curves, rewards, and feedback loops shape player attention? What is the difference between engaging design and exploitative design? What performance constraints determine whose devices can run a game? A lesson on profiling can become a lesson on access: inefficient software does not merely waste cycles; it may exclude students with older hardware.

The courses already contain a significant democratic impulse because they normalize public authorship. Students publish games, repositories, documentation, and web applications. This is a major shift from many school tasks, which disappear after grading. Public authorship can strengthen agency because it says: your work can exist outside the classroom, and you can contribute to the digital environment rather than only inhabit it. At the same time, public authorship requires ethical support. Students need to understand that publishing is not only a technical operation but a civic act. It involves copyright, accessibility, attribution, security, privacy, and responsibility to users. The course materials mention licenses, documentation, deployment, cloud environments, and safety practices, which creates an excellent foundation. The critical addition would be systematic reflection on power: platforms enable student publication, but they also set terms, collect data, rank visibility, and shape what counts as legitimate work.

Critical pedagogy would also ask how classroom talk is organized around these projects. A teacher can use the same lesson materials in either a liberating or a domesticated way. If students merely follow instructions and submit files, the curriculum's democratic promise is reduced. If students discuss why a project matters, compare designs, critique examples, share mistakes, and question the social meaning of their software, the curriculum becomes dialogical. For example, before building a highscore database, students might discuss whether ranking motivates or discourages different players. Before building a top-down shooter with AI, they might discuss how game genres represent conflict and agency. Before deploying a multiplayer tic-tac-toe game, they might discuss what kinds of communication, identity, and fairness a networked interaction requires. These conversations need not replace coding time; they can sharpen it.

The repeated emphasis on documentation can also be interpreted critically. README files, comments, user stories, diagrams, pitch presentations, and self-checklists are not merely school paperwork. They make technical work legible. Legibility is a condition of shared power. If only the original author can understand a program, then collaboration becomes dependence. If a project explains its purpose, setup, license, data model, and limits, then others can inspect, adapt, question, and improve it. This is why documentation belongs with democratic computing. It opens the artifact to community rather than enclosing it inside the student's private cleverness. The course sequence's repeated attention to documentation should therefore be defended as civic practice, not only as professional habit.

A critical lens also complicates the idea of creativity. The courses rightly foreground creativity through games, prototypes, UI, procedural generation, and project publication. Yet creativity in technological culture is often captured by novelty, speed, and market appeal. A critical version of the curriculum would expand creativity toward care, repair, accessibility, localization, and social usefulness. A student might be praised not only for adding a new mechanic, but for simplifying an interface so younger players can use it, translating terms, reducing file size, adding keyboard accessibility, explaining licensing clearly, or making error messages humane. Such work may be less spectacular than a new power-up, but it is ethically richer. It teaches students that technical creativity includes making systems more just and more understandable.

The local civic dimension could be strengthened even further. The course materials already align with broader transversal skills such as critical thinking, self-directed learning, collaboration, civic participation, and digital literacy. A critical pedagogy would connect those aims to local contexts. Students might analyze digital services used in their municipality, redesign a school process, build a small tool for a club, make a game about a local environmental issue, visualize anonymized school survey data, or create multilingual instructions for a community resource. Such projects would not abandon the existing technical goals; they would situate them. HTML, Python, databases, APIs, and Godot would become means of investigating and improving lived environments. Freire's idea of reading the word and the world is relevant here: students learn code most deeply when code helps them read the social world they actually inhabit.

Such localization would also protect students from the belief that meaningful technology is always produced somewhere else. A Latvian-language lesson about GitHub, Godot, PostgreSQL, or REST already performs a quiet act of cultural translation. Extending that translation into local problems would show that global tools can serve situated needs. The political question then becomes not whether students can imitate professional software culture, but whether they can appropriate its tools for communities they know.

Another critical issue is language. The lessons are written in Latvian but necessarily use many English technical terms: HTML, CSS, JavaScript, GitHub, README, IDE, input/output, Streamlit, Flask, REST API, JSON, PostgreSQL, GDExtension, Scene Tree, Node, object pooling, pathfinding, and so on. This bilingual reality is pedagogically rich but politically uneven. It can empower students to enter global technical discourse, but it can also make programming feel foreign, elite, or linguistically coded. A critical teacher should not erase English terminology, because it is part of the field. But the teacher should help students translate concepts, build glossaries, explain terms in their own words, and notice how language can gatekeep participation. The repeated use of Latvian explanations beside English technical vocabulary is therefore one of the curriculum's quiet strengths.

Access is another necessary concern. GitHub, VS Code, Godot, PostgreSQL, cloud deployment, and C++ build tools assume devices, internet, installation rights, and technical stability. The lessons often address practical setup and common errors, which is wise. But critical pedagogy asks whether every learner can participate under these assumptions. A student with an older computer, unstable home internet, or limited administrative permissions may experience "technical difficulty" as personal failure. The curriculum should make infrastructure visible. Teachers can reduce inequity by offering school-based setup time, offline alternatives, shared troubleshooting logs, paired work, and explicit validation that build errors are part of the toolchain rather than evidence of insufficient ability.

The game-centered arc also deserves critical examination. Games are powerful because they are motivating, interactive, culturally familiar, and technically rich. They make loops, state, collision, input, UI, audio, randomness, data structures, and feedback meaningful. Yet games are not universally motivating, and the gaming world carries its own exclusions around gender, language, genre, competition, and cultural taste. A critical implementation of this curriculum would widen the meaning of "game" and "software product." Students might create educational games, accessibility tools, simulations, civic data visualizations, local culture archives, or community problem-solving applications. The existing structure supports this expansion because it teaches transferable design, data, web, and deployment skills. The teacher's task is to prevent the genre from narrowing the learner's imagination.

Assessment is also political. If grading rewards only the final running product, students with prior experience, stronger devices, or more time outside school gain an unfair advantage. The course materials repeatedly foreground planning, testing, documentation, process, self-checklists, and project defense. This is important because it shifts value toward visible learning, not merely polished output. A critical pedagogy would go further by asking students to explain constraints, document failed attempts, describe help received, justify design tradeoffs, and reflect on user impact. Such assessment recognizes that software is made within conditions, not outside them.

Ultimately, the three-course sequence can be a strong vehicle for democratic technical literacy if it treats code as a way of reading and rewriting the world. The existing curriculum already moves students beyond passive consumption. It asks them to install tools, manage repositories, structure information, design interfaces, collect and analyze data, choose algorithms, respect licenses, build web systems, model objects, create AI behavior, and publish artifacts. The critical challenge is to keep asking: for whom, under whose conditions, with what consequences, and according to which values? Without those questions, the curriculum risks producing skilled adaptation to existing systems. With them, it can produce students who understand that technology is made by people, for people, and therefore open to ethical judgment and democratic redesign.

## Essay 3: The Phenomenology of Debugging, Care, Frustration, and Becoming a Programmer

The lived experience of learning programming is often underdescribed. Curriculum maps name topics, standards, and outcomes, but the learner meets programming as a sequence of felt encounters: the blank file, the first successful output, the confusion of a syntax error, the relief of a fixed bug, the embarrassment of a program that fails in front of someone else, the satisfaction of a game object finally moving, the anxiety of public presentation, and the strange pride of a link that can be sent to another person. A phenomenological and care-ethical reading of these three courses asks not only what students learn, but how programming appears to them as an experience. From this perspective, the most important repeated phrase across the lessons may not be any single technical term. It may be the recurring invitation to test, correct errors, and write a conclusion.

Programming education is full of breakdowns. Martin Heidegger's famous distinction between tools that are ready-to-hand and tools that become present-at-hand through breakdown helps illuminate novice programming. When code works, the student looks through it toward the task: the character moves, the score changes, the form submits, the JSON loads. When code fails, the tool becomes stubbornly visible: an indentation error, a missing semicolon, a wrong file path, a failed C++ compilation, a null reference, an unhandled input, a database connection error, an exported game that behaves differently from the editor. Good pedagogy does not eliminate breakdown. It teaches students to dwell with breakdown productively.

The course architecture repeatedly normalizes this experience. Many lessons include "Biežākās kļūdas" sections, testing tasks, debugging prompts, code examples, self-checklists, and project defense elements. This is not a minor convenience. It is a moral message to the learner: error is not shame; error is material for thinking. In Datorika 9, students must test HTML structure, CSS layout, media paths, JavaScript interactivity, DOM behavior, timers, and collision logic. In Programmēšana I, they must validate input, handle exceptions, organize files, read and write data, compare algorithms, connect Python to databases, design APIs, and deploy applications. In Programmēšana II, they must deal with the more intense friction of Godot-C++ integration: build tools, class registration, physics cycles, signals, object hierarchies, file access, profiling, and export. Each level increases the complexity of possible failure, but the curriculum's recurring structure can teach the student to meet failure with method rather than panic.

This is where care ethics becomes central. Nel Noddings argues that education is fundamentally relational, grounded in attentive response to the learner's need. In programming, care cannot mean removing difficulty, because difficulty is the medium through which understanding forms. Care means making difficulty inhabitable. A caring programming teacher helps the student distinguish between "I am bad at this" and "this system is giving me information I do not yet know how to interpret." The lesson design supports this distinction when it provides examples, error lists, time divisions, and reflection prompts. The teacher's live practice must complete it through tone, pacing, feedback, and the willingness to sit beside a student in the discomfort of not yet knowing.

The 70-minute work division found in many lessons is pedagogically interesting from this viewpoint. It frames time as a sequence of recoverable phases: first rebuild or understand a minimal prototype, then adapt it to the main task, then test and write a conclusion. For a novice, programming time can feel like undifferentiated pressure. A single unresolved bug can swallow the entire lesson psychologically. The time structure gives the learner a way to re-enter the task. Even if the second phase becomes difficult, the third phase still exists as testing and reflection. In other words, the lesson protects the student's sense that work has a shape. That shape matters for self-regulation.

The courses also stage a gradual transformation of identity. In Datorika 9, the student begins as someone who can make a web page and gradually becomes someone who can design, prototype, program, test, and publish a game. In Programmēšana I, the student becomes someone who can manage a repository, write Python, structure code, handle data, reason about algorithms, read documentation, use libraries, connect front-end and back-end, and deploy a networked project. In Programmēšana II, the student is invited into a more professional identity: a developer who understands engine architecture, C++ classes, AI, profiling, and release. This identity formation is not only cognitive. It is affective. Students learn what kind of frustration they can survive, what kind of precision they can sustain, what kind of public responsibility they can accept.

Phenomenologically, the transition from browser work to Python to Godot/C++ also changes the student's bodily relation to the computer. HTML and CSS give relatively immediate visual feedback; a refresh can reveal structure and style. JavaScript introduces invisible logic that becomes visible through interaction. Python in the terminal shifts attention to text, input, state, and program flow. Databases and APIs introduce systems that are not directly visible at all; the learner must infer them through responses, tables, routes, and JSON. Godot restores visual space but complicates it with editor hierarchies, compiled C++, physics loops, signals, and exported builds. The student therefore learns multiple modes of attention: visual inspection, textual precision, temporal reasoning, architectural imagination, and diagnostic inference.

The social texture of this experience is equally important. Programming can feel solitary even in a classroom full of students, because each screen displays a slightly different failure. The course sequence's use of GitHub, documentation, presentations, peer review, project defense, and shared artifacts can counteract that isolation. When students explain a bug to a peer, they often discover that their own understanding changes in the act of explanation. When they read someone else's README, they learn how documentation can either welcome or repel a future reader. When they present a prototype, they experience the vulnerability of showing unfinished work. These social moments are not extras after the real learning; they are part of the lived process by which a learner becomes able to participate in a community of practice.

Care also requires attention to pacing and identity differences. Some students will experience the game theme as freedom; others may experience it as pressure to perform in a culture they do not feel they belong to. Some will enjoy public presentation; others may find it exposing. Some will understand visual layout quickly but struggle with abstract loops; others will reason well about data but feel lost in UI design. A phenomenological teacher watches these differences closely. The same artifact can appear differently to different learners. A Godot scene tree might appear as a playful construction space to one student and as a maze of unfamiliar terms to another. A caring curriculum allows multiple routes to competence while still maintaining standards.

The courses' repeated movement from small examples to larger projects helps with this diversity because it gives learners multiple thresholds of success. A student may not complete every extension, but can still rebuild the prototype, implement the core mechanic, test one feature, or document a failed attempt. This matters because identity in programming is fragile early on. If the only recognized success is the fully polished final product, many students learn that they are outsiders. If partial but genuine progress is visible and valued, students can remain in the practice long enough to develop. The curriculum's structure supports such layered recognition, especially through basic tasks, main tasks, additional challenges, common errors, and self-checklists.

Common error sections deserve special attention as a form of anticipatory care. They tell learners that the teacher has seen this difficulty before and has prepared a path through it. This can reduce the loneliness of failure. When a lesson names likely mistakes with file paths, indentation, type conversion, collision layers, database commands, save files, or C++ build steps, it implicitly says: this problem belongs to the practice, not to your personal inadequacy. That message is philosophically important because shame narrows attention. A ashamed learner hides errors; a supported learner investigates them. The more advanced the course becomes, the more this anticipatory care matters, because advanced tools fail in ways that can look mysterious even to capable students.

This variety is one of the curriculum's deepest strengths. It prevents programming from being reduced to typing syntax. The student learns that code is sometimes a visible page, sometimes a terminal interaction, sometimes a repository history, sometimes a data model, sometimes a server endpoint, sometimes a moving object in a scene tree, and sometimes a performance profile. Each mode asks for a different kind of patience. A learner who struggles with one mode may find competence in another, and then bring that confidence back. For example, a student who finds terminal programs abstract may thrive when JavaScript changes a visible interface; a student who finds visual design distracting may discover satisfaction in database schemas or Big O comparisons.

However, the emotional load of such breadth should not be underestimated. Programmēšana II, in particular, introduces advanced tooling that can produce failure before the learner's own code is even conceptually involved. GDExtension setup, C++ compilation, version compatibility, editor registration, and export can all fail in opaque ways. The course materials wisely warn students not to rely only on editor clicking and repeatedly ask them to compile, check, and document results. Still, a phenomenological teacher would recognize that toolchain opacity can wound motivation. The student may feel excluded from a hidden professional world. The response should be a classroom culture where setup failures are collectively investigated, documented, and shared. The troubleshooting log becomes not only a technical resource but a sign that difficulty belongs to the community.

Reflection is therefore essential. The repeated instruction to write what works, test results, or conclusions should not be treated as an administrative afterthought. It is how experience becomes learning. Without reflection, a student may fix a bug by accident and remain unable to transfer the insight. With reflection, the student can name the pattern: the file path was wrong, the type conversion failed, the loop condition never changed, the collision layer was mismatched, the save file needed validation, the API route returned the wrong status. Naming transforms a private frustration into a shareable concept.

The curriculum's project defenses and presentations extend this transformation. To present a game specification, pitch an architecture, defend a project design, or explain a deployed web app is to turn making into speech. This matters because programmers often know more in action than they can articulate. The course rightly asks students to externalize their reasoning through README files, diagrams, user stories, pitch presentations, and self-checklists. From a care perspective, these artifacts also help teachers see the learner's process rather than only the final product. A program that is unfinished may still contain excellent planning, ethical awareness, or diagnostic reasoning. Conversely, a polished product may hide copied code or shallow understanding.

The philosophical conclusion is that debugging is not only a technical skill. It is an education in attention, humility, resilience, and relational trust. A student who learns to debug learns to listen to a system without immediately collapsing into self-judgment. A student who learns to test learns that reality has the right to answer back. A student who learns to document learns that future understanding deserves care. A student who learns to publish learns that an artifact enters a world of users. The three-course sequence is valuable because it repeatedly places the learner in this cycle of breakdown, repair, explanation, and renewed action. Its success will depend on whether classrooms preserve the humanity of that cycle. Programming is hard not because students are deficient, but because they are learning to think with unforgiving symbolic machines. Good pedagogy makes that encounter rigorous, honest, and bearable.

## Essay 4: Formation of Technical Judgment, Virtue Ethics, and Engineering Responsibility

A fourth reading of the three courses emphasizes formation. The question is not only "What knowledge does the curriculum transmit?" or "What projects does it enable?" but "What kind of technical person does it cultivate?" From an Aristotelian perspective, education is not merely the accumulation of information. It is the formation of habits, perceptions, judgments, and desires. Aristotle distinguishes between episteme, theoretical knowledge; techne, craft knowledge; and phronesis, practical wisdom. Computing education needs all three. Students need concepts, they need craft fluency, and they need judgment about what is worth doing, how to do it well, when to stop, and how to justify tradeoffs. The three courses, taken together, can be interpreted as a long apprenticeship in technical virtue.

Datorika 9 begins this apprenticeship with habits that are deceptively basic: organize files, name things clearly, use GitHub, structure HTML, style responsibly, optimize images and audio, consider target audiences, prototype interfaces, test JavaScript, debug game behavior, and publish a finished artifact. These are not simply beginner tasks. They are the early forms of professional discipline. A disorganized folder is not a small aesthetic flaw; it is the beginning of future confusion. An unclear interface is not merely ugly; it fails a user. A game that has not been tested does not yet deserve trust. A project published without reflection is not yet mature work. In virtue terms, the course introduces orderliness, care, patience, and responsibility.

Programmēšana I develops these virtues through increasing abstraction. The course begins with working environment, IDE setup, GitHub, cloud storage, synchronization, `.gitignore`, README files, Markdown, and licenses. This opening is philosophically important because it frames programming as a practice before it frames it as syntax. The student first learns to inhabit a disciplined workspace. Python basics then introduce precision with types, conversion, arithmetic, f-strings, style, comments, and input/output. Control structures and validation cultivate logical seriousness: the program must handle more than the happy path. Functions and modularity cultivate decomposition: a complex problem should be divided into meaningful responsibilities. Collections, files, JSON, CSV, data visualization, algorithms, databases, OOP, web integration, APIs, and deployment gradually build a larger field in which judgment is required.

Programmēšana II intensifies the demand for judgment by introducing systems in which superficial success can be misleading. A learner may place objects in Godot and see something move, but the course repeatedly insists that game rules and systems logic must be implemented in C++ GDExtension classes, not only through editor clicking. This insistence is an ethical and epistemic stance. It asks students to know what is actually causing behavior. The same logic appears in lessons on physics cycles, input maps, collisions, OOP, signals, data structures, AI state machines, A* pathfinding, object pooling, save/load architecture, procedural generation, profiling, debugging, UI signals, audio, and export. The student must learn to justify design choices rather than merely produce visible effects.

Virtue ethics helps name the habits being cultivated. Precision is one. Code punishes vagueness, but school can either turn that punishment into fear or into disciplined attention. The curriculum supports disciplined attention through repeated examples, common error sections, and testing tasks. Humility is another. Students learn that programs fail, inputs vary, files may not exist, databases may reject queries, APIs may return unexpected statuses, and exported builds may behave differently. Such lessons can teach epistemic humility: the developer does not assume; the developer checks. Courage is also present. Public presentation, GitHub publication, project defense, and playable artifacts require students to show work that may be imperfect. Temperance matters as well: students must resist adding features before the core works, resist visual polish before functionality, and resist copying without understanding.

The course sequence also develops what philosophers of knowledge call epistemic responsibility. A responsible knower does not only hold correct answers; they use appropriate methods for arriving at and revising those answers. In these courses, epistemic responsibility appears as version control, documentation, test cycles, validation, debugging, code style, modularity, data modeling, algorithmic complexity analysis, profiling, and security practice. When students compare linear and binary search, study Big O, or profile a game, they learn that claims about efficiency require reasons. When they design database tables with primary and foreign keys, they learn that data integrity is not an accident. When they deploy Flask applications and use environment variables, they learn that convenience must be balanced with security.

The curriculum's repeated final projects are crucial for forming judgment because projects require tradeoffs. A worksheet can be completed by satisfying local instructions; a project must hold together as a whole. In Datorika 9, a student building a full browser game must balance scope, visual design, JavaScript complexity, media assets, testing time, and publication. In Programmēšana I, projects such as memory cards, quiz games with highscores, browser tic-tac-toe, databases, and multiplayer tic-tac-toe require choices about data structures, user interface, persistence, and maintainability. In Programmēšana II, the student must choose how to structure classes, how to model states, how to store progress, how to optimize repeated objects, and how to prepare a build. These are not matters of syntax alone. They are occasions for phronesis, practical wisdom under constraint.

The inclusion of user stories, UX, target audience, surveys, presentations, licenses, and public deployment broadens technical virtue into civic and professional virtue. A technically correct program can still be careless if it confuses users, violates licenses, ignores accessibility, stores unnecessary data, or cannot be understood by collaborators. The courses repeatedly remind students that software is made for use, communication, and maintenance. This is particularly important in school, where students can otherwise internalize the idea that code exists only to satisfy a teacher. Here, the existence of README files, GitHub repositories, public pages, web apps, and game builds suggests a wider accountability.

The curriculum also teaches judgment through constraints. Beginners often imagine freedom as the absence of limits: add every feature, use every visual effect, store every piece of data, or build the most ambitious game possible. Engineering wisdom begins when constraints are taken seriously. Datorika 9 constrains students through browser technologies, media optimization, time planning, and publication requirements. Programmēšana I constrains them through Python syntax, validation, data formats, algorithmic complexity, relational schema design, and API contracts. Programmēšana II constrains them through C++ compilation, Godot's node model, physics timing, memory and performance considerations, save-file validity, and export targets. These limits are not enemies of creativity. They are the field within which meaningful creativity becomes possible.

Aristotle's idea of habituation is useful here. Students do not become careful by hearing one lecture about care. They become careful by repeatedly practicing careful actions until those actions become part of how they see a task. The three-course sequence repeats certain habits across many contexts: name files sensibly, test after changes, separate responsibilities, document decisions, consider the user, handle errors, justify choices, and publish responsibly. At first, these may feel like external teacher demands. Over time, they can become internal standards. A student begins to feel discomfort when a file is disorganized, when a function is too large, when input is unvalidated, or when a project lacks instructions. That discomfort is educationally valuable. It is the emerging conscience of craft.

The advanced course's attention to profiling and performance adds another layer to this formation. Performance is often misunderstood by novices as a purely technical or competitive concern: faster is better. A virtue-oriented view sees performance as responsibility under conditions. If a game wastes resources, drains batteries, stutters on modest devices, or fails in the browser, then the user's experience is harmed. Profiling teaches students to replace vague impressions with evidence. It also teaches restraint: optimize what matters, measure before claiming, and understand tradeoffs. The same can be said for security and deployment in Programmēšana I. Environment variables, cloud hosting, and API boundaries teach that a working demo is not the same as a responsible public system.

Code review, even in a modest classroom form, would be a natural companion to this virtue formation. The Datorika 9 lessons already mention analyzing another programmer's code, and the wider sequence contains many artifacts that can be reviewed: folder structures, functions, diagrams, SQL schemas, API routes, class hierarchies, state machines, and exported builds. Review teaches students to see quality beyond personal preference. It also teaches generosity: the purpose is to help the work become clearer, safer, and more usable. A virtue-oriented classroom would model review language carefully. Instead of "this is wrong," students might learn to say, "I cannot yet see where this value is validated," or "this function seems to have two responsibilities," or "the README does not tell a new user how to run the project." Such speech cultivates judgment and community at the same time.

The teacher's own modeling is decisive here. Students learn technical virtue not only from rubrics, but from watching an experienced person pause before a claim, test a hypothesis, admit uncertainty, read an error message slowly, and choose clarity over cleverness. When a teacher demonstrates debugging without embarrassment, cites documentation honestly, or revises an earlier design decision, the class sees that expertise is not omniscience. It is disciplined responsiveness. That is perhaps the most important professional habit a school course can transmit.

Still, formation-oriented education must avoid a narrow professionalism that values only industry-like productivity. There is a risk that GitHub, deployment, databases, APIs, Godot, C++, and cloud tools may make the classroom feel like a junior workplace rather than a space for intellectual growth. Virtue ethics offers a corrective: the aim is not to produce mini-employees but to cultivate people capable of excellent, responsible action. That means preserving time for explanation, wonder, alternative designs, and ethical questions. Why is this algorithm elegant? What does this interface ask of the user? When should a system not collect data? What makes a game fair? What does a license permit, and what does gratitude require beyond legality? These questions keep technical formation humanistic.

Assessment should therefore measure judgment as well as output. The course materials already include project defenses, self-checklists, planning documents, and criteria in some lessons. A virtue-oriented assessment system would make such elements central. Students should be assessed on whether they can explain their architecture, justify their data structures, identify risks, describe tests, interpret failures, document sources, and reflect on user needs. A broken but well-analyzed project may show more growth than a working project assembled without understanding. Conversely, a working project should still be questioned: Is it maintainable? Is it secure enough for its context? Are names clear? Are errors handled? Is the user respected?

This does not mean that correctness is unimportant. Virtue is not vague niceness. Technical virtue requires working artifacts. A program that never runs cannot fully serve its purpose. But correctness is only one dimension of excellence. A mature programmer seeks correctness with clarity, maintainability, fairness, humility, and care. The three-course arc is strong because it has room for all of these. It teaches students to make things, but also to structure, document, test, analyze, publish, and defend them. It introduces professional tools without reducing learning to tool use. It values creativity while repeatedly returning to constraints.

The philosophical conclusion is that the curriculum's deepest promise is character formation through technical practice. Students who complete the arc should not merely know some HTML, CSS, JavaScript, Python, SQL, Flask, Godot, and C++. They should have practiced habits of planning before building, testing before trusting, documenting before forgetting, asking who the user is, respecting sources, choosing structures with reasons, and repairing errors without despair. In a world increasingly organized by software, such habits are not optional extras. They are civic and intellectual virtues. The best defense of these courses, therefore, is not only that they teach programming. It is that they can teach students to become careful makers of shared digital worlds.
